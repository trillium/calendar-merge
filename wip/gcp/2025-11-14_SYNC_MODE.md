# Calendar Sync Architecture

**Date:** 2025-11-14
**Status:** Partially Implemented (needs fixes)

## Overview

The calendar sync system uses a **round-robin batch processing** approach to sync events from multiple source calendars to a target calendar. This architecture works identically in both local development and cloud deployment.

## Core Principle

**One calendar per request.** Each HTTP request to `/sync/trigger` processes ONE calendar and exits quickly. If more calendars need syncing, the function triggers itself again via HTTP, creating an asynchronous chain.

This approach:
- ✅ Keeps serverless functions short (<10 seconds)
- ✅ Avoids Cloud Function timeouts
- ✅ No Cloud Tasks billing required
- ✅ Same code locally and in production
- ✅ Respects API rate limits

---

## Round-Robin Sync Flow

### Initial Setup

When a user sets up calendar sync with 3 source calendars:

```
User → POST /calendars/watch (calendar A)
    → POST /calendars/watch (calendar B)
    → POST /calendars/watch (calendar C)
    → POST /sync/trigger (userId)
```

### Self-Triggering Chain

```
┌─────────────────────────────────────────────────────────────┐
│ POST /sync/trigger { userId }                               │
│                                                              │
│ 1. Get user's watches: [A, B, C]                           │
│ 2. Get current index: 0                                     │
│ 3. Sync calendar A (batch of events)                        │
│ 4. Update index: 1                                          │
│ 5. hasMore = (nextIndex !== 0) = true                      │
│ 6. Self-trigger: POST /sync/trigger { userId }             │
│ 7. Return immediately                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ POST /sync/trigger { userId }                               │
│                                                              │
│ 1. Get user's watches: [A, B, C]                           │
│ 2. Get current index: 1                                     │
│ 3. Sync calendar B (batch of events)                        │
│ 4. Update index: 2                                          │
│ 5. hasMore = (nextIndex !== 0) = true                      │
│ 6. Self-trigger: POST /sync/trigger { userId }             │
│ 7. Return immediately                                        │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│ POST /sync/trigger { userId }                               │
│                                                              │
│ 1. Get user's watches: [A, B, C]                           │
│ 2. Get current index: 2                                     │
│ 3. Sync calendar C (batch of events)                        │
│ 4. Update index: 0 (wrap around)                           │
│ 5. hasMore = (nextIndex !== 0) = false                     │
│ 6. No self-trigger (all calendars synced)                   │
│ 7. Return immediately                                        │
└─────────────────────────────────────────────────────────────┘
```

## Key Components

### 1. Round-Robin State (Firestore)

**Collection:** `syncState`
**Document ID:** `roundrobin_{userId}`
**Data:**
```json
{
  "currentIndex": 0  // Which calendar to process next
}
```

This persists between function calls, enabling the chain to continue.

### 2. Watch Documents (Firestore)

**Collection:** `watches`
**Document ID:** `{channelId}`
**Data:**
```json
{
  "userId": "115699614043593531056",
  "calendarId": "trillium@hatsfabulous.com",
  "targetCalendarId": "fe074822b3...@group.calendar.google.com",
  "channelId": "channel-1763101972719-ba2b535d2942aea5725d974606aae3e7",
  "paused": false,
  "syncToken": "COjZ5rmYkooDEOjZ5rmYkooDGAU=",
  "syncTokenUpdatedAt": 1731396789000,
  "syncState": {
    "status": "completed",
    "startedAt": 1731396776323,
    "completedAt": 1731396789000
  },
  "stats": {
    "totalEventsSynced": 8,
    "lastSyncTime": 1731396789000,
    "lastSyncEventCount": 8
  }
}
```

### 3. Batch Sync Service

**File:** `/gcp/src/services/batch-sync.service.ts`

**Function:** `batchSyncRoundRobin(userId: string): Promise<RoundRobinStatus>`

**Logic:**
1. Query all watches for user
2. Filter active watches (not paused, not currently syncing)
3. Get current index from `syncState/roundrobin_{userId}`
4. Process calendar at `activeWatches[currentIndex]`
5. Fetch events using `syncToken` (incremental sync)
6. Sync each event to target calendar
7. Update watch with new `syncToken`
8. Calculate next index: `(currentIndex + 1) % activeWatches.length`
9. Save next index to Firestore
10. Return `hasMore` flag

**Return Value:**
```typescript
{
  userId: string;
  currentIndex: number;        // Next calendar to process
  calendarsProcessed: 1;       // Always 1
  eventsProcessed: number;     // Events synced this batch
  hasMore: boolean;            // True if more calendars exist
}
```

### 4. Sync Controller

**File:** `/gcp/src/controllers/sync.controller.ts`

**Function:** `triggerBatchSync(req, res)`

**Logic:**
1. Extract `userId` from request body
2. Call `batchSyncRoundRobin(userId)`
3. Check `result.hasMore`
4. If `hasMore === true`: Self-trigger via HTTP (fire-and-forget)
5. Return result immediately

## hasMore Logic

The `hasMore` flag determines if the chain continues.

### Current Implementation (BROKEN)

```typescript
const hasMore = events.length === batchSize; // Checks if more EVENTS exist
```

**Problem:** This checks if the current calendar has more events, not if more calendars need syncing.

### Correct Implementation (NEEDED)

```typescript
const hasMore = nextIndex !== 0; // True until we wrap back to start
```

**Logic:**
- If `nextIndex = 1`: More calendars exist → `hasMore = true`
- If `nextIndex = 2`: More calendars exist → `hasMore = true`
- If `nextIndex = 0`: Wrapped around → `hasMore = false`

## Self-Triggering Mechanism

### Controller Implementation

```typescript
export async function triggerBatchSync(req: Request, res: Response): Promise<void> {
  const { userId, channelId } = req.body;

  try {
    if (userId) {
      log.info('Round-robin batch sync triggered', { userId });
      const result = await batchSyncRoundRobin(userId);

      // If more calendars exist, trigger next batch
      if (result.hasMore) {
        // Fire and forget - don't await
        fetch(`${APP_CONFIG.CLOUD_FUNCTION_URL}/sync/trigger`, {
          method: 'POST',
          headers: { 'Content-Type': 'application/json' },
          body: JSON.stringify({ userId })
        }).catch(err => log.error('Failed to trigger next batch', err));
      }

      res.status(200).json({ success: true, ...result });
    }
    // ... legacy channelId support
  } catch (error) {
    log.error('Error in batch sync', error, { userId, channelId });
    res.status(500).json({ error: 'Error processing batch sync' });
  }
}
```

### Why Fire-and-Forget?

```typescript
// ❌ BAD: Awaiting blocks the function
if (result.hasMore) {
  await fetch(...); // Function doesn't return until chain completes
}

// ✅ GOOD: Fire-and-forget allows quick return
if (result.hasMore) {
  fetch(...).catch(...); // Function returns immediately, chain continues async
}
```

## Local vs Cloud Behavior

### Local Development

**Environment:**
- Dev server on `http://localhost:8080`
- `CLOUD_FUNCTION_URL=http://localhost:8080`

**Flow:**
```
User triggers → localhost:8080/sync/trigger (calendar A)
                       ↓ self-trigger
                localhost:8080/sync/trigger (calendar B)
                       ↓ self-trigger
                localhost:8080/sync/trigger (calendar C)
                       ↓
                     Done
```

**Benefits:**
- Easy to debug (separate requests in logs)
- Each request completes quickly
- Can observe chain in real-time

### Cloud Deployment

**Environment:**
- Cloud Function at `https://region-project.cloudfunctions.net/calendar-sync`
- `CLOUD_FUNCTION_URL=https://region-project.cloudfunctions.net/calendar-sync`

**Flow:**
```
User triggers → Cloud Function (calendar A) → exits
                       ↓ self-trigger
                Cloud Function (calendar B) → exits
                       ↓ self-trigger
                Cloud Function (calendar C) → exits
                       ↓
                     Done
```

**Benefits:**
- Each function invocation is short (<10 seconds)
- No timeout issues
- Pay only for actual processing time
- Serverless scales automatically

## Sync Token Management

Each watch maintains a `syncToken` for incremental syncing.

### Initial Sync (No Token)

```typescript
const { events, nextSyncToken } = await listEvents(userId, calendarId, {
  maxResults: 2500,
  singleEvents: true,
  syncToken: undefined  // Fetch ALL events
});
```

### Subsequent Syncs (With Token)

```typescript
const { events, nextSyncToken } = await listEvents(userId, calendarId, {
  maxResults: 2500,
  singleEvents: true,
  syncToken: watch.syncToken  // Fetch only changes since last sync
});
```

**Google Calendar API Behavior:**
- Initial sync: Returns all events
- Token saved: `"COjZ5rmYkooDEOjZ5rmYkooDGAU="`
- Next sync: Returns only new/modified/deleted events
- New token returned: `"CPaR7bmYkooDEPaR7bmYkooDGAU="`

## Rate Limiting

Each event sync includes a delay to respect Google Calendar API quotas.

**Config:** `APP_CONFIG.RATE_LIMIT_DELAY_MS` (default: 1000ms = 1 second)

**Implementation:**
```typescript
for (let i = 0; i < events.length; i++) {
  const event = events[i];
  await syncEvent(userId, calendarId, event.id, targetCalendarId);

  if (i < events.length - 1) {
    await sleep(APP_CONFIG.RATE_LIMIT_DELAY_MS); // 1 second delay
  }
}
```

**Calculation:**
- 8 events per calendar
- 1 second delay between events
- ~8-10 seconds per calendar
- 3 calendars = ~30 seconds total (across 3 function calls)

## Error Handling

### Individual Event Failures

If a single event fails to sync, the batch continues:

```typescript
const result = await syncEvent(userId, calendarId, event.id, targetCalendarId);
if (result.success) {
  syncedCount++;
} else {
  failedCount++;
}
// Continue to next event
```

### Watch Failures

If an entire calendar sync fails:

```typescript
await db.updateDoc('watches', channelId, {
  'syncState.status': 'failed',
  'syncState.error': error.message,
});
```

The next time round-robin runs, it will skip failed watches unless they're reset.

### Self-Trigger Failures

If the self-trigger HTTP call fails:

```typescript
fetch(...).catch(err => log.error('Failed to trigger next batch', err));
```

The chain stops. User can manually re-trigger from the UI.

**Future Enhancement:** Implement retry logic or dead-letter queue.
